# AWS Multi-Environment Infrastructure with Terraform

This is a sanitized interview project that demonstrates reusable Terraform modules and separate AWS environments. No company names, account IDs, real AMIs, IP allowlists, backend bucket names, state files, plan files, credentials, or resource tag blocks are included.

## What it creates

- A shared VPC with public and private subnets
- Bastion, Grafana, internal tooling, and an Application Load Balancer in the shared environment
- Separate development, staging, and production VPCs
- VPC peering from each application environment to the shared VPC
- Amazon EKS clusters and managed node groups
- PostgreSQL on Amazon RDS with credentials generated by Terraform and stored in AWS Secrets Manager
- EC2-based search/Elasticsearch hosts
- IAM roles and policies, including IRSA/OIDC integration
- Centralized remote state in Amazon S3
- Optional Atlantis automation for plan and apply workflows

## Deployment order

Deploy in this order because the application environments read outputs from the shared remote state:

```text
shared → dev → staging → prod
```

## Configure and run

For each environment, copy the example files and replace only the placeholder values with values from your AWS account:

```bash
cd terraform/envs/shared
cp backend.hcl.example backend.hcl
cp terraform.tfvars.example terraform.tfvars

terraform init -backend-config=backend.hcl
terraform fmt -check -recursive
terraform validate
terraform plan -var-file=terraform.tfvars
terraform apply -var-file=terraform.tfvars
```

Repeat for `dev`, `staging`, and `prod` after the shared environment has been applied.

Do not commit `backend.hcl`, `terraform.tfvars`, `.terraform/`, state files, or saved plan files.

## Atlantis flow

`atlantis.yaml` defines one project per environment. A pull request triggers `terraform plan`; an approved and mergeable pull request can run `terraform apply`. The environment dependencies ensure that shared infrastructure is handled first.

## Interview explanation

> I created reusable Terraform modules for networking, compute, EKS, RDS, IAM, and VPC peering. I separated shared, development, staging, and production environments so each environment has its own state and configuration. The shared environment provides common services such as bastion access and monitoring, while the application environments create their own VPC, EKS cluster, database, and search server. Terraform remote state passes shared resource IDs to the dependent environments. RDS credentials are generated securely and stored in Secrets Manager. Atlantis can run plan and apply from pull requests with approval controls.

## Security notes

- The repository contains no AWS credentials or private keys.
- RDS passwords are not exposed as Terraform outputs.
- Backend and environment values are represented by `.example` files.
- EKS API, SSH, monitoring, and database access should be restricted to trusted CIDRs and security groups.
- The AWS Load Balancer Controller IAM policy contains AWS API actions such as `CreateTags`; these are permissions required by the controller, not environment-specific resource tag values.

## Important note about tags

All Terraform `tags`, provider `default_tags`, tag variables, and tag outputs were removed as requested. In a real production environment, tags are normally recommended for ownership, cost allocation, automation, and governance. EKS subnet discovery may also require specific Kubernetes subnet tags depending on the load balancer setup.
